# PRD: 행사안전 체크 MCP

## 1. 문서 개요

| 항목 | 내용 |
|---|---|
| 제품명 | 행사안전 체크 |
| 영문명 | Event Safety Check |
| 신규 저장소명 | `event-safety-check-mcp` |
| 원본 저장소 | `korea-mice-safety-agent` |
| 개발 방식 | 맥미니의 최신 원본을 복사해 별도 저장소로 개발 |
| 목표 플랫폼 | PlayMCP, 이후 Kakao Tools |
| 제품 버전 | `0.1.0` |
| 우선 목표 | AGENTIC PLAYER 10 예선 제출 |

---

## 2. 핵심 결정

`korea-mice-safety-agent`의 법령·조례·위험요인·베뉴 온톨로지와 검증 로직은 유지한다.

단, 공모전용 제품에서는 다음을 변경한다.

1. MICE 전문 용어를 일반 행사 언어로 전환한다.
2. 일반 사용자가 자주 여는 학교 축제, 플리마켓, 야외공연, 전시, 사내행사를 전면에 배치한다.
3. 기존 도구 전체를 노출하지 않고 핵심 도구 6개만 제공한다.
4. 로컬 파일 저장과 상태형 현장 운영 기능은 제외한다.
5. Stateless Streamable HTTP 서버로 제공한다.
6. 원본 저장소는 수정하지 않는다.

## 4. 문제 정의

소규모 행사 주최자는 다음 정보를 한꺼번에 판단하기 어렵다.

- 행사 규모에 따라 무엇을 준비해야 하는가
- 임시무대, 전기, LPG, 식음료 사용 시 무엇이 달라지는가
- 안전관리계획과 비상대응 문서가 필요한가
- 어느 법령·조례·시설 규정을 먼저 확인해야 하는가
- 빠뜨린 안전 항목은 무엇인가

일반 AI는 법령 적용 범위를 과장하거나 존재하지 않는 의무를 생성할 수 있다. 검색 결과도 행사 조건과 관계없이 전국의 법령과 조례를 나열하는 경우가 많다.

행사안전 체크는 행사 조건을 구조화해 **적용 후보, 비적용 항목, 필요한 문서, 위험 통제, 확인할 기관**을 근거와 함께 제공한다.

## 7. 핵심 사용자 시나리오

### 학교 축제

> 학교 운동장에서 800명 규모 축제를 해. 무대와 푸드트럭이 있고 임시전기를 사용할 예정이야. 무엇부터 준비해야 해?

> 성수동 야외 플리마켓에 2,000명이 올 예정이야. 천막 40개와 LPG 푸드트럭이 있어. 신고와 안전 준비를 점검해줘.

### 야외공연

> 공원에서 5,000명 규모 공연을 진행해. 무대, 발전기, 야간 공연, 도로 일부 통제가 있어. 안전관리계획 초안을 만들어줘.

### 실내 전시

> 코엑스에서 100개 부스 규모 전시를 열어. 설치·철거와 전기 사용 규정을 확인해줘.

### 기존 계획 검수

> 이 안전관리계획에서 빠진 내용과 과도하게 적용한 법령을 구분해줘.

## 10. MCP 도구 구성

총 6개만 PlayMCP에 노출한다.

### 10.1 `assess_event_safety`

행사 조건을 바탕으로 안전 준비 항목을 진단한다.

입력:

- 행사명
- 행사 유형
- 장소 또는 관할 지역
- 예상 최대 인원
- 실내·실외
- 도로 사용
- 임시 구조물
- 임시 전기
- LPG·가스
- 식음료
- 공연
- 설치·철거
- 고소작업
- 중량물
- 개인정보·CCTV

출력:

- 입력 조건 요약
- 우선 실행 항목
- 적용 후보
- 비적용 항목
- 필요한 문서
- 핵심 위험요인
- 추가로 확인할 정보
- 근거와 데이터 기준일

### 10.2 `create_event_safety_plan`

행사 안전관리계획 초안을 생성한다.

출력 구성:

1. 행사 개요
2. 역할과 책임
3. 인파·동선 관리
4. 무대·부스·임시구조물
5. 전기·화기·LPG
6. 설치·철거 작업자 안전
7. 소방·피난
8. 응급의료
9. 식음료 안전
10. 개인정보·CCTV
11. 비상연락 및 대응
12. 사전 제출·협의 체크리스트
13. 행사 전·당일·종료 후 실행 항목

파일을 생성하지 않고 Markdown과 `structuredContent`를 반환한다.

### 10.3 `review_event_safety_plan`

사용자가 제공한 안전관리계획을 검수한다.

검수 항목:

- 필수 정보 누락
- 행사 조건과 맞지 않는 계획
- 과도하게 적용된 법령
- 근거가 없는 의무 표현
- 빠진 위험요인
- 담당자·기한 누락
- 비상대응 부족
- 추가 확인이 필요한 항목

판정은 다음 세 단계로 제한한다.

- 확인됨
- 보완 필요
- 관할기관 확인 필요

`적법`, `허가 가능`, `법적으로 완전함` 같은 확정 표현은 사용하지 않는다.

### 10.4 `search_event_venues`

KOPIS 오프라인 공연시설 인덱스를 검색한다.

검색 조건:

- 시설명
- 지역
- 시설 분류
- 주소
- 키워드

결과는 기본 5개, 최대 10개로 제한한다.

### 10.5 `get_event_venue_rules`

거점 베뉴의 시설·안전규정을 조회한다.

반환 항목:

- 반입·하역
- 바닥하중
- 천장고
- 전기
- 소방·피난
- 부스·리깅
- 식음료
- 금지행위
- 제출 문서
- 출처와 확인일

### 10.6 `get_event_risk_controls`

특정 위험요인에 대한 예방조치와 현장 확인사항을 제공한다.

지원 위험:

- 군중 밀집
- 출입구 병목
- 임시 구조물
- 무대·트러스·리깅
- 임시 전기
- 추락·고소작업
- 중량물
- 화기·LPG
- 식중독
- 소방·피난
- 응급환자
- 개인정보·CCTV
- 기상악화

## 11. 도구 메타데이터

모든 도구에 다음을 지정한다.

```json
{
  "title": "사용자용 한글 제목",
  "readOnlyHint": true,
  "destructiveHint": false,
  "idempotentHint": true,
  "openWorldHint": false
}
```

도구 설명은 영문을 기본으로 작성하되 서비스명을 영문·국문으로 병기한다.

예시:

```text
Assesses event safety requirements using Event Safety Check(행사안전 체크), an offline Korean event-law, venue-rule, hazard-control, and checklist knowledge base.
```

서버명과 도구명에는 `kakao` 문자열을 사용하지 않는다.

## 12. 입력 정규화

사용자 표현을 내부 행사 유형으로 변환한다.

| 사용자 표현 | 내부 유형 |
|---|---|
| 학교 축제, 대학 축제 | `festival` |
| 플리마켓, 장터 | `festival`, `outdoor_event` |
| 버스킹, 야외공연 | `performance`, `outdoor_event` |
| 전시, 박람회 | `exhibition` |
| 포럼, 세미나, 컨퍼런스 | `conference` |
| 푸드트럭 행사 | `food_event`, `outdoor_event` |
| VIP 행사 | `vip_event` |

필수 정보가 부족하면 한 번에 최대 3개 질문만 요청한다.

우선 질문:

1. 예상 최대 인원
2. 실내·실외와 장소
3. 임시무대·전기·가스·식음료 여부

## 13. 출력 UX

첫 응답은 다음 순서로 제공한다.

### 1. 행사 조건 요약

사용자가 입력한 조건과 시스템이 추론한 조건을 구분한다.

### 2. 이것부터 하세요

최대 5개의 실행 항목만 우선 표시한다.

각 항목에 포함:

- 해야 할 일
- 이유
- 기한 또는 확인 시점
- 근거 유형
- 확인 기관

### 3. 필요한 문서

- 법정 의무 후보
- 베뉴 제출 문서
- 권장 체크리스트
- 관할기관 확인 필요

### 4. 주요 위험요인

최대 5개를 위험 이유와 통제방법으로 표시한다.

### 5. 적용하지 않은 항목

사용자가 불필요한 규정을 준비하지 않도록 비적용 사유를 표시한다.

### 6. 주의사항

- 데이터 기준일
- 최신 원문 확인 필요
- 법률·허가 판단을 대체하지 않음

## 14. 응답 데이터 계약

모든 핵심 도구는 Markdown과 구조화 데이터를 함께 반환한다.

```json
{
  "eventProfile": {},
  "attentionLevel": "basic | enhanced | high_review",
  "topActions": [],
  "requiredDocuments": [],
  "riskControls": [],
  "applicableCandidates": [],
  "notApplicable": [],
  "missingInputs": [],
  "sources": [],
  "meta": {
    "service": "Event Safety Check(행사안전 체크)",
    "version": "0.1.0",
    "dataDate": "YYYY-MM-DD",
    "disclaimer": "..."
  }
}
```

`attentionLevel`은 법적 적합성 점수가 아니다. 입력 조건에 따라 추가 검토 필요도를 표현하는 용도로만 사용한다.

## 15. 기술 아키텍처

### 권장 구조

```text
src/
  domain/
    기존 온톨로지 조회 및 판단 로직
  adapters/
    event-input-adapter.ts
    event-result-adapter.ts
  tools/
    assess-event-safety.ts
    create-event-safety-plan.ts
    review-event-safety-plan.ts
    search-event-venues.ts
    get-event-venue-rules.ts
    get-event-risk-controls.ts
  presenters/
    action-summary.ts
    plan-markdown.ts
    review-markdown.ts
  server/
    mcp-server.ts
    http.ts
    health.ts
  config/
    limits.ts
    version.ts
```

### 구현 원칙

- 기존 도메인 로직은 가능한 한 수정하지 않고 어댑터로 감싼다.
- 기존 30여 개 내부 도구는 삭제하지 않아도 되지만 PlayMCP registry에는 등록하지 않는다.
- 시작 시 JSON 온톨로지를 메모리에 로드하고 인덱스를 생성한다.
- 요청 처리 중 네트워크를 호출하지 않는다.
- 요청 처리 중 파일을 쓰지 않는다.
- 동일 입력은 동일 결과를 반환한다.
- 로그는 표준 오류로만 출력한다.

## 16. PlayMCP 서버 요구사항

- Streamable HTTP 전송
- 공개 Endpoint 경로: `/mcp`
- 상태 확인 경로: `/health`
- `0.0.0.0` 바인딩
- `PORT` 환경변수 지원
- Stateless 동작
- MCP Inspector 테스트
- 지원 프로토콜 범위 충족
  - 최소 `2025-03-26`
  - 최대 `2025-11-25`
- 요청 본문 크기 제한
- 계획서 검수 입력 최대 50,000자
- 결과 크기 최소화
- 오류 시 원시 스택이나 내부 경로를 사용자에게 노출하지 않음

### 성능 목표

- 일반 조회 평균 100ms 이내 목표
- 계획 생성·검수 p99 3초 이내
- 콜드 스타트 이후 모든 데이터 조회는 메모리 기반
- 시설 검색 결과 최대 10개
- 법령·위험요인 결과는 관련도 상위 항목만 반환

## 19. 테스트 시나리오

최소 다음 10개를 자동 테스트한다.

1. 800명 학교 축제, 무대·푸드트럭·임시전기
2. 2,000명 플리마켓, 천막·식음료·LPG
3. 5,000명 야외공연, 도로 통제·야간 운영
4. 300명 실내 전시, 부스 설치·철거
5. 800명 컨퍼런스, 참가자 등록·CCTV
6. 40명 실내 워크숍, 위험시설 없음
7. 주최자 없는 10,000명 운집 예상
8. VIP 행사, 경호·보안검색
9. 공연 없는 행사에서 공연법 비적용 확인
10. 식음료 없는 행사에서 식품·LPG 규정 비적용 확인

### 검수 기준

- 관련 없는 전국 조례를 우선 결과로 내지 않는다.
- 비적용 규정을 의무처럼 표시하지 않는다.
- 존재하지 않는 법령명과 조문을 생성하지 않는다.
- 동일 입력의 결과가 결정적이다.
- 모든 핵심 판단에 근거 또는 `관할기관 확인 필요` 상태가 있다.
- 사용자가 이해하기 어려운 MICE 전문 용어를 첫 응답에서 사용하지 않는다.

## 20. 공모전 제출용 설명

### 서비스명

**행사안전 체크**

### 짧은 설명

> 학교 축제, 플리마켓, 야외공연, 전시 등 행사 조건을 입력하면 필요한 안전조치, 준비 문서, 위험요인과 확인할 규정을 알려주는 행사 운영 안전 비서입니다.

### 긴 설명

> 행사를 준비할 때 주최자는 인파, 임시무대, 전기, LPG, 식음료, 소방, 개인정보와 지역별 규정을 동시에 확인해야 합니다. 일반 검색은 행사와 관계없는 법령까지 나열하고, 생성형 AI는 존재하지 않는 의무를 만들 수 있습니다. 행사안전 체크는 한국 행사 안전 법령·조례·베뉴 규정·위험 통제 온톨로지를 기반으로 행사 조건에 맞는 준비 항목만 선별합니다. 필요한 문서와 우선 행동을 알려주고, 안전관리계획 초안을 생성하거나 기존 계획의 누락과 과잉 적용을 검수할 수 있습니다.

### 대표 예시 질문

- “학교 축제에 800명이 올 예정이야. 무대와 푸드트럭이 있는데 무엇을 준비해야 해?”
- “성수동 야외 플리마켓에서 LPG와 임시전기를 사용해. 필요한 신고와 안전 문서를 점검해줘.”
- “이 행사 안전관리계획에서 빠진 항목과 불필요하게 적용한 규정을 구분해줘.”

## 22. 완료 정의

다음을 모두 만족해야 완료로 본다.

- 원본 저장소에 변경사항이 없다.
- 신규 저장소가 독립적으로 빌드된다.
- Docker 이미지가 단독 실행된다.
- `/health`가 정상 응답한다.
- `/mcp`에서 6개 도구만 노출된다.
- 모든 도구에 annotations가 있다.
- MCP Inspector에서 도구 목록과 호출이 정상이다.
- PlayMCP에서 Endpoint 정보 불러오기가 성공한다.
- 대표 질문 10개가 오류 없이 수행된다.
- 법령·조례·베뉴 근거가 출력된다.
- 비적용 항목을 의무로 표시하지 않는다.
- 원본 문서와 비공개 데이터가 컨테이너에 포함되지 않는다.
- 서비스 설명에서 일반 사용자가 MICE 지식 없이도 사용법을 이해할 수 있다.
